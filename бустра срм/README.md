Для сборки локального окружения разработчика в Docker необходимо.

1. Склонировать к себе проект на рабочее устройство

`git clone git@ggitlab.boostra.tech:boostra/crm.git`

2. Переключиться на ветку develop

`git checkout develop`

3. Получить у коллег файлы конфигурации и поместить их сюда:

- `./config/config.php`
- `./config/.env`

4. Запустить Docker-Compose файл из корня проекта.

`docker-compose up -d`

5. Если необходимо пересобрать контейнер выполняем следующую команду:

`docker-compose up -d --build`

## Процесс разработки

### Структура веток

Проект использует git flow модель с двумя основными ветками:

- **`develop`** - предрелизная ветка для разработки
- **`master`** - продакшн ветка с стабильным кодом

### Создание рабочей ветки

Все рабочие ветки создаются от ветки `develop`:

```bash
# Обновляем develop до актуального состояния
git checkout develop
git pull origin develop

# Создаем новую ветку с соответствующим префиксом
git checkout -b feature/CS-123  # для новых фич
git checkout -b fix/CS-456      # для исправлений багов
git checkout -b hotfix/CS-789   # для критических исправлений
```

### Рабочий процесс

1. **Разработка в своей ветке:**
   ```bash
   git add <changed files>
   git commit -m "Описание изменений"
   git push origin feature/CS-123
   ```

2. **Создание Merge Request:**
   - После завершения работы создайте MR своей ветки с веткой `develop`
   - **НЕ мерджите ветку напрямую** - используйте только MR через GitLab
   - Дождитесь code review и аппрува

3. **Тестирование:**
   - Для тестирования используйте тестовые контуры
   - В GitLab в разделе Pipelines можно задеплоить любую ветку на тестовый контур
   - После успешного тестирования и code review делается мердж в `develop`
   - Мердж `develop` → `master` для продакшена выполняют старшие разработчики

### Работа с исправлениями

**Если ветка уже в develop и требует доработки:**
```bash
git checkout feature/CS-123
git commit -m "fix: исправление после ревью"
git push origin feature/CS-123
```

**Если требуется критическое исправление (hotfix):**
```bash
git checkout develop
git checkout -b hotfix/CS-123
git commit -m "hotfix: критическое исправление"
git push origin hotfix/CS-123
```

**Важно:** Разработчики не работают с веткой `master` напрямую. Ветка `develop` всегда должна быть актуальной относительно `master`.

## Кэширование (Redis)

Для оптимизации производительности в системе используется Redis. Основной метод для работы — `\Caches::wrap`, который автоматически проверяет наличие данных в кэше и сохраняет их при отсутствии.

### Список кэшируемых данных

| Объект / Метод | Время (TTL) | Описание ключа | Что именно кэшируется |
| :--- | :--- | :--- | :--- |
| **Заявки (Orders)** | **30 сек** | `orders:count:[hash]` | Количество заявок с учетом активных фильтров. |
| **Черный список (Blacklist)** | **10 мин** | `blacklist:in:[userId]` | Статус нахождения пользователя в черном списке. |
| **Менеджеры (Managers)** | **1 час** | `managers:all` | Полный список всех менеджеров системы. |
| **Права доступа** | **1 час** | `permissions:[role]` | Список разрешений для конкретной роли. |
| **Организация (ID)** | **1 час** | `organization:[id]` | Полный объект данных организации по её ID. |
| **Организация (ИНН)** | **1 час** | `organization_id_by_inn:[inn]` | ID организации, найденный по её ИНН. |
| **ИНН по сайту** | **1 час** | `inns_by_site_id:[siteId]` | Список ИНН, привязанных к конкретному сайту. |
| **ИНН по заказу 1С** | **1 час** | `inns_by_order_1c_id:[uid]` | Список ИНН, связанных с UID заказа из 1С. |
| **ИНН по User ID** | **1 час** | `inns_by_user_id:[userId]` | Список ИНН организаций, связанных с ID пользователя. |
| **ИНН по User UID** | **1 час** | `inns_by_user_uid:[uid]` | Список ИНН организаций, связанных с UID пользователя. |

### Особенности реализации
- **Сохранение типов:** Кэш использует нативную PHP-сериализацию (`serialize`/`unserialize`), что гарантирует полное сохранение типов данных, включая вложенные объекты и массивы.
- **Отказоустойчивость:** При недоступности Redis система автоматически переключается на прямые запросы к БД.
- **Тестирование:** Для проверки работы кэша используется скрипт `script/cache_test.php`.
