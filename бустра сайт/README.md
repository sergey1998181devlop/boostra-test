Для сборки локального окружения разработчика в Docker необходимо.

1. Склонировать к себе проект на рабочее устройство

`git clone git@gcib0d3c8bstra.boostra.ru:boostra/boostra.git`

2. Переключиться на ветку develop

`git checkout develop`

3. Получить у коллег файлы конфигурации и поместить их сюда:

- `./config/config.php`

4. Добавить секцию `[site]` в config.php для работы с настройками:

```ini
[site]
; ID сайта из таблицы s_sites для загрузки специфичных настроек
; Если не указан или пустой - используются только глобальные настройки (site_id IS NULL)
; Примеры: boostra, soyamarket, rubl, и т.д.
site_id = ;
```

**Подробнее о системе настроек:** см. документацию в CRM `crm/docs/site-settings.md`

5. Запустить Docker-Compose файл из корня проекта.

`docker-compose up -d`

6. Если необходимо пересобрать контейнер выполняем следующую команду:

`docker-compose up -d --build`

## Доступ к проекту

- Проект обычно доступен по адресу `http://localhost:8088/`.
- Если проект не открывается — проверьте что контейнер запущен (`docker ps`), включите отображение ошибок в `index.php` и проверьте логи в директории `logs/`.

## Процесс разработки

### Структура веток

Проект использует git flow модель с двумя основными ветками:

- **`develop`** - предрелизная ветка для разработки
- **`master`** - продакшн ветка с стабильным кодом

### Создание рабочей ветки

Все рабочие ветки создаются от ветки `develop`:

```bash
# Обновляем develop до актуального состояния
git checkout develop
git pull origin develop

# Создаем новую ветку с соответствующим префиксом
git checkout -b feature/CS-123  # для новых фич
git checkout -b fix/CS-456      # для исправлений багов
git checkout -b hotfix/CS-789   # для критических исправлений
```

### Рабочий процесс

1. **Разработка в своей ветке:**
   ```bash
   git add <changed files>
   git commit -m "Описание изменений"
   git push origin feature/CS-123
   ```

2. **Создание Merge Request:**
    - После завершения работы создайте MR своей ветки с веткой `develop`
    - **НЕ мерджите ветку напрямую** - используйте только MR через GitLab
    - Дождитесь code review и аппрува

3. **Тестирование:**
    - Для тестирования используйте тестовые контуры
    - В GitLab в разделе Pipelines можно задеплоить любую ветку на тестовый контур
    - После успешного тестирования и code review делается мердж в `develop`
    - Мердж `develop` → `master` для продакшена выполняют старшие разработчики

### Работа с исправлениями

**Если ветка уже в develop и требует доработки:**
```bash
git checkout feature/CS-123
git commit -m "fix: исправление после ревью"
git push origin feature/CS-123
```

**Если требуется критическое исправление (hotfix):**
```bash
git checkout develop
git checkout -b hotfix/CS-123
git commit -m "hotfix: критическое исправление"
git push origin hotfix/CS-123
```

**Важно:** Разработчики не работают с веткой `master` напрямую. Ветка `develop` всегда должна быть актуальной относительно `master`.
